## 1. What Trenador Is

Trenador is a members-only AI fitness coaching web app for **PowerhouseSoFlo** (Powerhouse Gym — Miami & Fort Lauderdale). It runs as a single production tenant, but the schema is multi-tenant-ready: every table carries a `tenant_id`, and queries are scoped through `APP_CONFIG.tenantId`.

**Who uses it**
- **Members** — gym clients. They sign up, complete a 5-step onboarding (including picking a coach), and then get AI chat, the workout library, a personal workout builder, and human coach messaging. Access is gated by a trial-then-paid subscription.
- **Admins** — a single-page admin dashboard for managing members, coaches, and workouts, and for replying to member messages as a coach. An admin is any member with `is_admin = true` (set manually in the database).
- **Coaches** — coaches exist as data: a member is assigned one and can message them, but **there is no coach-facing portal**. Admins act on coaches' behalf. Coaches and workouts are created and published from the admin dashboard.

**Business model** — a Stripe subscription (7-day free trial, then paid) gates member access. The app is currently in **Stripe test mode**; live mode is blocked on the business EIN. Personal-training packages are sold out-of-band.

## 3. Next.js 16 Quirks (these bite people constantly)

- **`exactOptionalPropertyTypes: true`** means you cannot explicitly pass `undefined` to an optional property or column. Use spread patterns or `value ?? undefined`. Drizzle inserts use `value ?? undefined` for optional columns, and `?? null` only when the column is nullable and you actually mean null.
- **Route params are Promises.** A page receives `params: Promise<{ id: string }>`, so you must `const { id } = await params`. The same applies to `searchParams`.
- **Caching uses the new directives.** Use `'use cache'` with `cacheTag()` / `cacheLife()`, not `unstable_cache`. Invalidate with `revalidateTag()` / `revalidatePath()`.
- **There is no `middleware.ts`.** Next 16 renamed it; session refresh lives in **`src/proxy.ts`**. Note that the proxy only refreshes the Supabase session — it does **not** enforce auth. Auth is enforced at the layout/action level.
- The dev server runs on Webpack (`next dev --webpack`).

## 5. What's Built vs. Not

**Built**
- **Auth:** email/password, Google OAuth, magic link, and password reset (Supabase Auth). Signup is self-serve, and admins can also invite members (`adminInviteUser`).
- **Onboarding:** a single 5-step flow — first/last name, birth year, gender, weight, and a coach picker. It sets `members.assigned_coach_id` and writes a (currently thin) `intake_submissions` row.
- **AI chat:** streaming over SSE, a 30-messages-per-day rate limit (Upstash), Haiku auto-titling, prompt caching, thread pinning, starter shortcuts, and a mobile slide-out sidebar with a FAB.
- **Workout library:** browse, filter, and search the gym's published workouts.
- **Personal builder ("My Workouts"):** create, edit, and delete workouts; set per-exercise targets; build from scratch or Save/Remix from the library.
- **Workout logging + history:** fully built, but **hidden from the primary nav**. The routes exist under `/log`; they're just not linked yet.
- **Coach messaging:** a member ↔ coaching-team thread, with an admin inbox that can reply, edit, and delete messages.
- **Coaches:** a member-facing `/coaches` directory, plus admin coach management. Note the per-coach `system_prompt` column exists but is **not yet wired into the chat**.
- **Profile:** avatar upload (Supabase Storage, served via signed URLs) and personal info.
- **Subscribe / paywall:** Stripe Checkout plus a webhook, in test mode.
- **Admin dashboard:** members (verify, suspend, invite, assign coach), coach CRUD, workout CRUD with publish/unpublish, and the messaging inbox.

**Not built / not wired**
- Stripe **live mode** (blocked on EIN).
- Privacy and Terms pages (owned by the product/legal owner). The `terms_acceptances` table exists, but acceptance isn't recorded yet.
- A coach-facing portal.
- Read-only mode for lapsed members — gated layouts currently just redirect to `/subscribe`.
- Injecting each coach's `system_prompt` into the AI (the column is there; nothing reads it yet).
- RAG / knowledge base (pgvector).
- Push notifications and Twilio SMS.
- Observability: Sentry (errors) and PostHog (product analytics).
- A staging environment (in progress).
- Automated tests.

## 6. Coding Conventions

- **Comments:** only when something isn't obvious, and then explain *why*, not *what*. The codebase puts a one-line rationale above genuinely tricky blocks (RLS intent, cache markers, ownership checks) and nothing elsewhere. Match that.
- **No mock or placeholder data.** Wire to real queries or leave the feature unbuilt. Don't fabricate seed values inside components.
- **Respect the strict compiler.** Honor `exactOptionalPropertyTypes` (spread or `?? undefined`) and `noUncheckedIndexedAccess` (array access is `T | undefined`, so guard it).
- **Server actions** start with `'use server'`; client components start with `'use client'`; server-only libraries import `'server-only'` at the top.
- **Authenticate every mutation.** Call `getAuthenticatedMember()` first. Guard writes with `assertActiveSubscription(member)`. Admin actions call a local `requireAdmin()`.
- **Scope every query by tenant** (`APP_CONFIG.tenantId`). Don't write cross-tenant queries.
- **Verify ownership** on member-scoped reads and writes by filtering on `memberId` — never trust an ID from the client without checking it belongs to the caller (see `getMessages`, `getMemberWorkout`, `seedLogSession` for the pattern).
- **Drizzle:** schema lives in `src/db/schema/*` and is re-exported from `index.ts`. Generate migrations with `npm run db:generate`, review the SQL, and apply manually. Inspect data with `npm run db:studio`.
- **Units:** logged set weights are in **pounds** (`workout_log_sets.weight_lbs`), but personal-workout targets are stored in **kilograms** (`target_weight_kg`) and converted for display. Convert carefully when comparing the two.
- **Styling:** Tailwind v4 utilities plus the tokens in `globals.css`. Compose class names with `cn()` from `lib/utils`, and prefer the existing `components/ui/*` primitives.

## 7. Auth & Subscription Flow (in one paragraph)

Supabase Auth issues a cookie-based session, which `proxy.ts` refreshes on navigation. A database trigger (`handle_new_user`, a `SECURITY DEFINER` function pinned to `search_path=public` and not callable over REST) creates a `members` row whenever an `auth.users` row is inserted. `getAuthenticatedMember()` maps the auth user to that members row, redirecting to `/login` if it's missing. Member surfaces live under `(member)/`, whose layouts call `requireActiveSubscription()`. That helper uses `isMemberActive()`: **admins and members with `subscription_status = 'active'` always pass; everyone else passes for 7 days from `created_at`**, and otherwise is redirected to `/subscribe`. Subscription state lives directly on the `members` row (`subscription_status`, `stripe_customer_id`) — there's no separate subscriptions table and no cron. Stripe Checkout leads to `/api/stripe/webhook`, which writes status changes straight onto `members`.

One thing to keep in mind: the app talks to Postgres directly through `postgres-js` (via `DATABASE_URL`), and **that connection bypasses RLS**. So the server-action guard layer is the primary authorization boundary; RLS protects the Supabase client surface (REST, Realtime, Storage) as defense in depth.

## 8. Git Workflow & Safety Rules (read this — non-negotiable)

These rules exist because several people work in this repo, including browser-based Claude Code sessions. A single bad push can break production for everyone.

- **Never commit or push to `main`.** `main` is production. Do not target it with a commit or push under any circumstances. (Branch protection is still being set up, so do not rely on GitHub to stop you — self-enforce this.)
- **Never run `git add`, `git commit`, or `git push` on your own.** Do not stage, commit, or push automatically — not even to a feature branch. Finish the work, show what changed, and then **ask the maintainer** whether they want you to create a branch, commit, and push. Only proceed after they explicitly say yes.
- **When asked to push:** create a descriptively named feature branch off `main`, commit with a clear message, push the branch, and open a PR. The commit must never land directly on `main`.
- **Merging is a human's job, not yours.** Every change lands through a PR that requires at least one review approval and passing CI (see below) before a person merges it.
- Follow a read-confirm-plan-approve loop before any large change, and prefer small, reviewable PRs.

**CI expectation:** PRs are gated on continuous integration — lint, type-check, and build must pass before merge. CI is not fully set up yet (it's on the roadmap), so until it is, run `npm run lint` and `npm run build` locally and confirm they're clean before proposing a PR.

## 9. Session Gotchas (read before you start)

1. **`CLAUDE.md` and `AGENTS.md` are committed to the repo** — they are no longer gitignored. Every session and contributor gets this context automatically. If you have personal or machine-specific notes, put them in `CLAUDE.local.md`, which is still gitignored.
2. **`README.md` is still the create-next-app boilerplate** — it isn't project documentation, so don't rely on it.
3. **The AI prompt assumes data onboarding no longer collects.** The system prompt says the AI "knows your goals, history, and body," but onboarding now captures only name, birth year, gender, weight, and coach. The richer health intake (goals, injuries, medical history, experience) is gone. Don't build AI features that assume that data exists until the intake question is resolved.
4. **`member_codes` is dead schema.** The table and migrations exist, but the member-code verification flow is no longer wired into onboarding. Treat it as unused unless it's intentionally revived.
5. **Stripe is in test mode**, and the webhook has no idempotency/replay guard yet — don't assume live keys or exactly-once delivery.

## 10. Keeping These Docs Current (your responsibility)

This project is documented by a small set of files. Treat them as living documents: **when a code change makes one of them wrong or incomplete, update that doc in the same PR as the code change.** Docs that drift from the code are worse than no docs.

**The doc set and what each one owns**

| Doc | File | Owns |
|---|---|---|
| CLAUDE.md | `CLAUDE.md` (this file) | How to work in the repo: stack, quirks, structure, conventions, workflow, gotchas. |
| PRD | `docs/PRD.md` | Product scope and priorities — what we're building, why, and the acceptance criteria. The "what/why." |
| Architecture | `docs/ARCHITECTURE.md` | System shape, data flow, data model, and the reasons behind key technical decisions. The "how it fits together." |
| Design System | `docs/DESIGN_SYSTEM.md` | Design tokens, typography, shape/motion, and component conventions as implemented in `globals.css` and `components/ui`. |
| Roadmap | `docs/ROADMAP.md` | The live task tracker — launch-readiness checklist, CI/CD and environment setup, and the post-MVP backlog. The "what's left." |

**When to update which doc**
- A feature moves **in or out of MVP scope** (e.g. something previously post-MVP is now needed for launch): update the **PRD** scope and priorities, and reflect it in the **Roadmap**. Scope decisions live in the PRD; the task to do the work lives in the Roadmap.
- A **migration changes the data model**, or an auth/subscription/AI flow changes: update the **Architecture** (data model + flows) and, if it changes how contributors work, **CLAUDE.md**. Keep the "synced to migration NNNN" line current.
- A **design token, font, or component convention** changes: update the **Design System**.
- A checklist item is **done**, or new work is discovered: check it off or add it in the **Roadmap** — don't turn the PRD into a task list.
- The **stack, conventions, or workflow** change: update **CLAUDE.md**.

**How to update**
- Bump the `Version:` (or "last synced") line at the top of any doc you edit. Small factual corrections are a patch bump; new sections or scope changes are a minor bump; a structural rewrite is a major bump.
- Reference docs by their role, not a hardcoded filename — the authoritative version is the line inside each doc, and filenames should stay stable in the repo (`docs/PRD.md`, not a version-suffixed name) so links and this table don't break.
- Propose doc updates as part of your plan, and include them in the same PR. If `CLAUDE.md`/`AGENTS.md` are still gitignored, remember they need `git add -f` (or un-ignore them) to actually be committed.
- If a doc and the code disagree, the **code wins** — fix the doc to match.
